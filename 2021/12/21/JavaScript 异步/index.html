事件循环

事件队列

开始进入主线程的执行栈时会把遇到的任务分为同步任务和异步任务两部分，同步任务会直接进入主线程依次执行，遇到异步任务挂起到 Event Table，当异步任务完成并返回结果后，将异步任务的回调放入事件队列中。

宏微任务

为什么区分宏微任务

为了可以插队

事件队列中又划分为宏微任务两个部分。
当执行栈中的任务执行完毕时，会先检查微任务队列。如果存在回调，全部压入执行栈执行。如果没有，执行宏任务队列中的下一个回调。
不同的异步任务，根据执行优先级被分为两类：macro（宏任务）和 micro（微任务）。

  • 常见的 macro-task 比如：setTimeout、setInterval、 setImmediate 等。
  • 常见的 micro-task 比如: process.nextTick、new Promise().then(回调)、MutationObserver(html5 新特性) 等。

Node 环境

在 node 中，事件循环表现出的状态与浏览器中大致相同。不同的是 node 中有一套自己的模型。node 中事件循环的实现是依靠的 libuv 引擎。
外部输入数据–>轮询阶段(poll)–>检查阶段(check)–>关闭事件回调阶段(close callback)–>定时器检测阶段(timer)–>I/O 事件回调阶段(I/O callbacks)–>闲置阶段(idle, prepare)–>轮询阶段
​

这些阶段大致的功能如下：
​

timers: 这个阶段执行定时器队列中的回调如 setTimeout() 和 setInterval()。
I/O callbacks: 这个阶段执行几乎所有的回调。但是不包括 close 事件，定时器和 setImmediate()的回调。
idle, prepare: 这个阶段仅在内部使用，可以不必理会。
poll: 等待新的 I/O 事件，node 在一些特殊情况下会阻塞在这里。
check: setImmediate()的回调会在这个阶段执行。
close callbacks: 例如 socket.on(‘close’, …)这种 close 事件的回调。
