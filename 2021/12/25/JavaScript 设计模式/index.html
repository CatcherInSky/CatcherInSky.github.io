设计原则

​

设计模式

单例模式

指一个类仅有一个实例，并提供一个访问它的全局访问点。简单的讲就是一个类只能创建一个实例，即使创建了多个实例，这些实例都是相同的。不会出现被覆盖的问题，保证只有一个实例，并提供全局访问。

vuex

vue 的状态管理模式，采用集中式存储管理应用的所有组件的状态，利用一个全局的 Store 存储应用的状态，提供 api 供用户做读写操作。不同的页面都能访问该 Store 对象，获取的值也是一样的，实现了不同页面数据保持同步。其本质上的实现就是通过单例模式。vuex 的实现是单例模式的经典应用。
​

构造器模式

构造器属性赋值给对象的过程封装，确保了每个对象都具备这些属性，确保共性的不变，同时将 个别属性取值操作开放，确保了个性的灵活。
​

class

​

策略模式

处理需要多个 switch…case…或者 if…else…的场景。策略模式的核心思想就是：寻找业务中的不变的部分和变化的部分，把不变的隔离，可变的封装起来。策略模式的目的就是将算法的使用和算法的实现分离开。这种就是策略模式。
一个基于策略模式的代码至少由两部分组成，一是策略类，策略类封装具体的算法，并负责具体的计算过程。二是环境类Context，Context 接受客户的请求，随后把请求委托给某策略类做处理，Context 需要维持对某策略对象的引用。
​

访问者模式

babel

转换器会遍历 AST 树，找出自己感兴趣的节点类型, 再进行转换操作.如果每个插件自己去遍历 AST，对不同的节点进行不同的操作，维护自己的状态。这样子不仅低效，它们的逻辑分散在各处，会让整个系统变得难以理解和调试。所以转换器操作 AST 一般都是使用访问器模式，由这个访问者(Visitor)来 ① 进行统一的遍历操作，② 提供节点的操作方法，③ 响应式维护节点之间的关系；而插件(设计模式中称为‘具体访问者’)只需要定义自己感兴趣的节点类型，当访问者访问到对应节点时，就调用插件的访问(visit)方法。
​

观察者模式


promise 有三个状态，pending,resolved,rejected 状态，.then()方法监听 resolved 状态，.catch()监听 rejected 状态。也是一种观察模式

事件委托

所有的 DOM 事件监听机制都是观察者模式。如事件委托
​

发布订阅模式

发布订阅模式主要是解决多个对象之间的通信问题，其中一个对象的行为发生变化可能会导致一个或者多个其他对象的行为也发生改变。一般具有发布者、订阅者、调度中心三个角色。
和观察者模式本质上是一样的，区别在于，发布订阅者模式中发布者和订阅者之间存在第三者，也就是调度中心，发布者和订阅者之间没有耦合，统一由调度中心做信息内容的调度，发布者只负责往调度中心推送，订阅者只需要接受调度中心的通知获取信息。发布订阅模式做到了对象之间的解耦。
​

常用设计模式

MVC 架构思想

mvc 架构设计的提出目的就是将视图界面和业务逻辑分离，从而达到代码的可拓展性、复用性、可维护性等，降低相应的业务耦合度
​

view 组合模式

组合模式将对象组合成树形结构，以表示“部分-整体”的层次结构，组件化思想。组合模式让客户端可以统一单个对象和组合对象的处理逻辑(递归遍历)。
​

view+model 观察者模式

model 发生变化，view 会被通知做相应的视图更新，但是 model 和 view 不会产生直接关联，当 model 中的数据发生变化，不需要 model 主动将数据的变化的消息推送给 view，而是通过 controller 来处理
​

view + controller 策略模式

view 和 controller 是一对多的关系，controller 是 view 的一个策略，controller 对于 view 来说是可以替换的
